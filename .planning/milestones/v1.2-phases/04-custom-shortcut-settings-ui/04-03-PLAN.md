Completely replace the existing SettingsForm.cs. The comment "// Phase 4 redesigns this form" in the current file confirms this is expected. Do NOT attempt to incrementally patch — write the new form from scratch.

**Key design decisions from research:**
- 2x2 corner grid: TableLayoutPanel (2 cols × 2 rows), each cell a GroupBox
- Each GroupBox: action ComboBox + "Record" Button + KeyRecorderPanel
- Monitor selector: ComboBox populated once at open from `Screen.AllScreens` (never in Paint)
- ComboBox bound by `Screen.DeviceName` (not by index — prevents ordering instability, Pitfall 6)
- Escape guard: override `ProcessCmdKey` to suppress Escape→CancelButton when any panel is recording (Pitfall 2)
- Win key note: small Label in the corner grid area — "Note: Win key shortcuts are not recordable"

**Form layout (approximate, use AutoScaleMode.Dpi):**
- ClientSize: ~400 × 460 (wider to accommodate corner grid)
- Monitor selector group at top (hidden when single monitor)
- Corner grid group in middle (~340 × 200)
- Detection params group (ZoneSize, DwellDelay) below grid
- System group (StartWithWindows) below detection
- Save/Cancel buttons at bottom

**Corner grid cell layout (per GroupBox):**
- GroupBox title: "Top Left", "Top Right", "Bottom Left", "Bottom Right"
- Row 1: Label "Action:" + ComboBox (items: "Disabled", "Task View", "Show Desktop", "Action Center", "Custom Shortcut")
- Row 2: Label "Shortcut:" + KeyRecorderPanel (display-only when action != Custom; enabled for Record) + Button "Record"
- Show/hide Record button and KeyRecorderPanel based on whether action is Custom

**Corner ordering in the 2x2 grid (matches physical screen):**
- [0,0] Top Left, [1,0] Top Right (column 0, row 0 and column 1, row 0)
- [0,1] Bottom Left, [1,1] Bottom Right

**Form data model (internal, not exposed as public properties like old form):**
- `_screens`: `Screen[]` — captured once in constructor
- `_selectedDeviceName`: `string` — tracks selected monitor
- `_pendingMonitorConfigs`: `Dictionary<string, MonitorCornerConfig>` — accumulates edits across monitor switches
- Each monitor's data is saved to `_pendingMonitorConfigs` when switching monitors

**Public API for HotSpotApplicationContext:**
```csharp
// Returns all monitored configs the user edited (may include unchanged monitors).
public Dictionary<string, MonitorCornerConfig> GetMonitorConfigs() => _pendingMonitorConfigs;
public int SelectedZoneSize => (int)_zoneSizeInput.Value;
public int SelectedDwellDelay => (int)_dwellDelayInput.Value;
public bool SelectedStartWithWindows => _startupCheckBox.Checked;
```
Remove the old `SelectedCorner` property — no longer needed.

**Monitor loading logic:**
- On open: load `Screen.AllScreens` into `_screens`. If `settings.MonitorConfigs` has an entry for a screen's `DeviceName`, populate from it; otherwise create a new `MonitorCornerConfig` with defaults (all Disabled).
- Clone entries into `_pendingMonitorConfigs` so the caller's settings object is not mutated until Save.
- When monitor selector changes: save current grid state to `_pendingMonitorConfigs[currentDeviceName]`, then load grid from `_pendingMonitorConfigs[newDeviceName]`.

**ProcessCmdKey override (Pitfall 2):**
```csharp
private bool _anyPanelRecording;

protected override bool ProcessCmdKey(ref Message msg, Keys keyData)
{
    if (_anyPanelRecording && keyData == Keys.Escape)
        return false;  // let Escape reach the focused KeyRecorderPanel
    return base.ProcessCmdKey(ref msg, keyData);
}
```
Set `_anyPanelRecording = true` in each panel's `StartRecording` (wire via a lambda on the Record button click handler). Reset in `ShortcutRecorded` and `RecordingCancelled` handlers.

**Record button wiring per corner:**
```csharp
recordButton.Click += (_, _) =>
{
    _anyPanelRecording = true;
    panel.StartRecording();
};
panel.ShortcutRecorded += (vks, displayText) =>
{
    _anyPanelRecording = false;
    actionCombo.SelectedIndex = /* index of "Custom Shortcut" */;
    panel.IdleText = displayText;
    panel.Invalidate();
    // Store in working config for this corner
    UpdatePendingCustomShortcut(corner, vks, displayText);
};
panel.RecordingCancelled += () =>
{
    _anyPanelRecording = false;
    panel.Invalidate();
};
```

**Action ComboBox items (use exact strings, map to enum on save):**
- "Disabled" → CornerAction.Disabled
- "Task View" → CornerAction.TaskView
- "Show Desktop" → CornerAction.ShowDesktop
- "Action Center" → CornerAction.ActionCenter
- "Custom Shortcut" → CornerAction.Custom

Show/hide the KeyRecorderPanel and Record button: listen to `actionCombo.SelectedIndexChanged` and set `panel.Visible = recordButton.Visible = (selectedAction == CornerAction.Custom)`.

**Manual layout only** (no designer file) — consistent with existing SettingsForm pattern. Keep `SuspendLayout` / `ResumeLayout` pattern.

**Note on Win key:** Add a small italic label below the corner grid: "Win key shortcuts (Win+Tab, Win+D, etc.) use the built-in actions above." This covers the inability to record Win key combinations.
