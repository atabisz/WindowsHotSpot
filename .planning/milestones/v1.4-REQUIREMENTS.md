# Requirements Archive: WindowsHotSpot v1.4 Window Drag Anywhere

**Archived:** 2026-05-05
**Status:** All requirements complete ✅

## Milestone v1.4 Requirements

### Hook Infrastructure

- [x] **HOOK-01**: `HookManager` exposes a suppression mechanism so consumers can prevent mouse button events from reaching target windows
- [x] **HOOK-02**: Suppression applies only to `WM_LBUTTONDOWN` / `WM_LBUTTONUP` — `WM_MOUSEMOVE` is never suppressed

### Window Drag

- [x] **DRAG-01**: User can hold Ctrl+Alt and left-click-drag anywhere on a window to move it
- [x] **DRAG-02**: The topmost window under the cursor is moved (not necessarily the foreground window)
- [x] **DRAG-03**: The initiating click is not forwarded to the target application while in drag mode
- [x] **DRAG-04**: The window position tracks the cursor delta in real time (smooth drag)
- [x] **DRAG-05**: Drag ends cleanly on mouse button release
- [x] **DRAG-06**: Maximized windows are ignored — drag does not apply and the click is forwarded normally

### Modifier Guard

- [x] **GUARD-01**: Only LCtrl+LAlt triggers drag — RCtrl+LAlt (AltGr) does not trigger
- [x] **GUARD-02**: If Ctrl or Alt is released mid-drag, drag is cancelled and the window stays at its current position

### Wiring

- [x] **WIRE-01**: `WindowDragHandler` is instantiated and wired to `HookManager` events in `HotSpotApplicationContext` alongside `CornerRouter`
- [x] **WIRE-02**: `WindowDragHandler` is disposed cleanly with the rest of the application components

## Future Requirements (deferred)

- Right-click or middle-click drag variants — left-button only for simplicity
- Per-app allow/deny list for drag — different product surface
- Resize-by-drag on window edges — separate Win32 concern, different UX pattern
- Drag on UAC-elevated windows — UIPI blocks SetWindowPos; no workaround without elevation match

## Out of Scope

- Right-click or middle-click drag — left-button only; adding more buttons multiplies edge cases without clear user need
- Window resize by drag — different Win32 patterns (WM_SIZING/SWP_NOSIZE vs move); separate concern
- Per-app profiles for drag — different product category
- Drag on UAC-elevated windows — UIPI blocks SetWindowPos from non-elevated process; detected and skipped cleanly
